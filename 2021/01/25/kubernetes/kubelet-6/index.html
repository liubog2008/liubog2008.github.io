Kubernetes 源码分析之 kubelet(六)

Version

v1.18.2

Container Manager

/pkg/kubelet/cm 下定义了一系列容器相关的功能，这个 cm 就是 container manager 的缩写(也有可能是 cgroups manager 的缩写?)

本次文章就是主要介绍一下 /pkg/kubelet/cm 下的功能，并简单聊一聊 container manager 和 container runtime 的差异(为什么 kubelet 需要实现一个 container manager)

以下是 cm 下的各个文件功能的简介

  • cpumanager: 实现 cpu 相关的资源管理
  • cpuset: cpuset 相关的 cgroups 操作
  • devicemanager: device plugin 相关的管理
  • topologymanager: kubelet 可能需要感知一些硬件的拓扑结构(e.g. NUMA)
  • cgroup_manager: cgroup 相关的操作
  • container_manager: 整个功能的入口
  • node_container_manager: 部分 node 级别的 cgroups 的管理(e.g. node allocatable)
  • pod_container_manager: 部分 pod 级别的 cgroups 的管理
  • qos_container_manager: qos 相关的 cgroups 管理

Why container manager?

先来说说为什么 kubelet 除了 container runtime 外还需要 cm 下的一系列功能。

如果有看过 kubelet 管理的节点下的 cgroups 结构，会发现其实 kubelet 并不是把创建容器的请求转发给 container runtime 就结束了，而是会对整个 cgroups 结构一定程度上进行直接的管理。

而对此进行管理有以下几个原因：

  1. 为了实现 kubernetes 定义的 QoS 模型
  2. pod 的层级结构导致的一些 pod 层面的资源限制和统计问题
  3. 为了提高节点稳定性，限制一些 container runtime 并不管理的资源(e.g. docker daemon)
  4. …

总之，kubelet 为了更好的进行节点上的资源管理，定义了 container manager 这么一个模块。

QoS

我们知道 pod 是 kubernetes 提出的概念，因此 container runtime 中是不会对 pod 进行任何 cgroups 设置的。这会导致以下情况

  1. 当某个容器没有设置 request 和 limit，但使用了超额资源后，需要整个节点承担代价

为了提高整个节点的稳定性，kubelet 定义了广为人知的 3 个级别的 QoS

  1. Guaranteed
  2. Burstable
  3. BestEffort

而为了实现这 3 个级别的 QoS，kubelet 定义了如下的 cgroups hierarchy

Guaranteed

首先 Guaranteed 是 pod 中所有容器的 request 都等于 limit 时的 QoS，此时的 cgroups 结构也比较简单，直接将每个容器的限制相加就行了

Burstable

Burstable 定义的是 request 小于 limit 的情况，Burstable 相比 Guaranteed 的 cgroups hierarchy 有一个显著的变化就是增加了一个 burstable 的 cgroup，所有的 burstable pod 都在这个 cgroup 之下

BestEffort

BestEffort 定义的是没有设置 request 和 limit 的情况，和 burstable 一样，有一个 besteffort 的 cgroup。

Design doc

详细的设计可以再看看 kubernetes 的 design doc

这个模块主要是处理系统保留资源用的，最终会创建一个 node allocatable 的 cgroup /kubepods，然后所有的 pods 都会定义在这个 cgroup 内

最后

container manager 这个模块主要是有助于了解 kubelet 对管理节点资源做了哪些努力，本次文章中主要介绍了 kubelet 的 cgroups hierarchy 相关的内容，没有详细介绍 cpumanager，devicemanager，topologymanager 相关的内容。了解这部分内容可以先去看看详细的 design doc，目前 kubernetes 的 design doc 有一部分在 community 里(较早期的一些设计，包括一些设计原则)，还有一些在 enchancement 里。
